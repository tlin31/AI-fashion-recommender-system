What Diagram C Shows
The fashion-specific additions built on top of Gorse. Gorse handles the recommendation engine — fashion-recommend adds:
  • LLM trait extraction from user text
  • Recommendation explanations
  • AI style chat
  • Auth, likes, comments
fashion-recommend API
A Gin-based HTTP layer that sits in front of Gorse. It adds social features (likes, comments), session auth, and LLM capabilities. Gorse knows nothing about these — they are purely fashion-recommend concerns.
Path A — The Key Bridge
This is how the fashion layer feeds intelligence into Gorse. User text → LLM → structured labels → written to user.Labels in Gorse. The CTR model then uses these labels as features. No fashion code runs inside Gorse — it just reads labels.
Trait Extraction
traits/extractor.go maps free Chinese/English text to structured English Gorse labels:
  • "极简风" → minimalist
  • "大地色" → earth_tones
  • "日常穿搭" → casual
Labels are normalized to English so Gorse's index is consistent.
Path B — Explanation Layer
Gorse returns a ranked list. fashion-recommend optionally calls the LLM to generate a human-readable explanation: "We picked these because you like minimalist earth-tone styles." Gorse is unaware this explanation was generated.
Path C — Fully Isolated
AI chat is entirely within fashion-recommend. It never calls Gorse. It maintains a conversation history in PostgreSQL and calls the LLM directly. It can optionally pull item data from Gorse to ground responses.

Auth Layer
fashion-recommend adds session-based auth that Gorse doesn't have. auth/ package manages sessions. All API routes require a valid session cookie before proxying to Gorse.
Gin Router Structure
api/server.go:96 mounts:
  • /api/* → Gorse proxy
  • /api/chat → AI service
  • /api/like → social handler
  • /images → static files
Label Schema
Labels written to Gorse must match the schema the CTR model was trained on. fashion-recommend normalizes all extracted traits to English before writing — Chinese text stays in PostgreSQL comments only.
Additive Labels
Trait extraction is additive — new labels merge with existing ones. A user who comments "I love streetwear" on Monday and "prefer neutral tones" on Friday accumulates both traits. Labels are never reset automatically.
Explanation is Optional
LLM explanation in Path B only fires if the client requests it (via a flag in the request). It adds 100–500ms latency. When disabled, Path B is as fast as Diagram B's Path A.
Chat is Fully Isolated
Path C never touches Gorse. It reads user.Labels from PostgreSQL (written by Path A) to ground the LLM prompt, but the conversation itself is entirely managed by fashion-recommend's database/ package.
The Key Insight
fashion-recommend and Gorse communicate through data, not code. No fashion-specific code runs inside Gorse. The entire integration is: fashion writes labels + feedback → Gorse reads them. This is a clean separation of concerns.
The architectural principle of the fashion-recommend layer: fashion-recommend does not modify or extend Gorse — it feeds it. Every piece of intelligence the fashion layer generates (extracted style traits, feedback events, item labels) is written to Gorse's standard data interfaces. Gorse remains a generic recommendation engine that happens to be getting unusually rich, LLM-structured inputs. The two systems are decoupled by design: you could replace either one without touching the other, as long as the label and feedback schemas remain consistent.
