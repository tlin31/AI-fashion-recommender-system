Use Case Flow — The Learning Cycle in Plain Words

How a single user action eventually results in a better recommendation list for every user

1
A user does something
fashion-recommend

Everything starts with a user action — a like, a purchase, a comment, or even just a view. The fashion-recommend API layer receives it.

  • A like or purchase is recorded as a positive feedback event. It tells the system "this user found this item relevant."
  • A comment does double duty: it's recorded as feedback, and the LLM extracts structured style traits from the text ("casual," "minimalist," "earth tones") and writes them back to the user's label set in Gorse. These labels will later feed directly into CTR model features.
  • A view without engagement is a weak signal — recorded as a negative or neutral event that the CTR model will learn to down-weight.
Why this matters: The user doesn't fill out a style quiz. Every natural interaction passively teaches the system what this person likes — and the LLM-powered trait extraction turns unstructured comments into structured signals the ranking model can act on.
↓
All feedback is written to the shared PostgreSQL data store
2
Master wakes up and reads everything
Gorse Master

On a configurable schedule (e.g., every 60 minutes), the Master node's task loop fires. It can also be triggered manually from the admin dashboard.

  • It streams all users, all items, and all feedback events from the database in batches.
  • It checks whether the dataset has actually changed since the last run. If nothing new arrived, it skips the entire training cycle — no wasted compute.
  • Each feedback event is classified: positive (the user engaged meaningfully) or negative (they saw it but didn't engage).
  • The data is split: 80% for training, 20% for testing model quality.
The Master is the only node that reads raw training data. Server and Worker never touch the feedback history directly.
↓
Training tasks run one at a time — they share the same loaded dataset
3
Master trains two models and builds three indexes
Gorse Master

Five tasks run sequentially. Each one produces outputs that downstream nodes will consume:

  • Collaborative Filtering model — learns a latent vector for every user and every item purely from who liked what. Users with similar taste end up with similar vectors. The output is stored as a binary in the blob store.
  • CTR Prediction model (AFM) — learns to estimate how likely a specific user is to click on a specific item, using structured features: the user's style labels (written by the LLM trait extractor), the item's category and attribute labels, and the latent embeddings from CF. The output is also stored in the blob store.
  • Item-to-Item similarity index — computes how similar every pair of items is, using tag overlap, embedding distance, or LLM-generated descriptions. Stored in the cache so workers can do fast similarity lookups.
  • User-to-User similarity index — finds users with overlapping engagement histories. Stored in cache.
  • Non-personalized scores — evaluates every item for trending, popularity, and recency using configurable expressions. Also stored in cache for fallback recall.
The fashion-recommend connection: The Master doesn't call any fashion-specific code. But the data it trains on — specifically user.Labels (style traits) and item.Labels (categories, colors, occasions) — was written by the fashion layer. The CTR model implicitly learns "users with 'minimalist' label tend to click items with 'monochrome' and 'clean-cut' labels."
↓
New model IDs written to the meta store — a monotonically increasing Unix timestamp
4
Trained artifacts land in shared storage
Shared Storage

Three storage systems get updated simultaneously:

  • Blob store receives the serialized CF and CTR model binaries. Each is keyed by its Unix timestamp ID. Old models are garbage-collected once the new ones are confirmed active.
  • Cache store receives updated similarity scores (item-to-item, user-to-user) and non-personalized lists. Workers can now do faster and more accurate recall lookups immediately.
  • Meta store records the new model IDs, their evaluation scores (AUC, precision, recall), and the hyperparameters used. This is what Worker nodes poll to detect an update.
The model ID being a Unix timestamp is an elegant design choice — any node can detect "this is newer than what I have" with a single integer comparison. No distributed lock or version negotiation needed.
↓
Worker polls master every few seconds — sees the new model ID
5
Worker detects the update and pulls the new models
Gorse Worker

Each Worker runs a continuous background loop that asks the Master: "what's the current model version?"

  • When the Worker sees a model ID higher than what it currently holds, it signals its internal pull loop.
  • The pull loop downloads the new CF and CTR model binaries from the blob store and deserializes them into memory.
  • Once loaded, it immediately triggers a fresh recommendation run — it doesn't wait for the next scheduled tick. This means recommendations are refreshed as soon as a new model is available.
  • Multiple workers operate independently on different user shards (consistent hashing). They all converge to the same model version within one poll interval.
↓
Recommendation pipeline runs for all users in this worker's shard
6
New recommendations are generated and cached
Gorse Worker

For each active user with a stale recommendation cache, the Worker runs the full pipeline:

  • Recall — assembles a candidate pool from five sources using the now-updated CF model, item-to-item similarity scores, and non-personalized lists.
  • Rank — reorders candidates using the updated CTR model (or LLM reranker if configured). Because the CTR model was just retrained on the latest feedback — including the style traits written by the LLM extractor — it reflects the user's most recent interactions.
  • Cache write — the final ranked list is written to the cache store, keyed by user ID.

The Server reads from this cache on every GET /recommend/{userId} request. From the Server's perspective, it just reads a pre-computed list — it has no idea a model was just retrained.

↓
Next request to /recommend returns the fresh list
↺
The user sees better recommendations — and the cycle repeats
Closed Loop

The user now sees a recommendation list shaped by their own past behavior and the behavior of similar users. If the recommendations are better, they engage more — which generates more feedback, which improves the next model, which further improves recommendations.

The key insight of the learning cycle: there is no human in the loop. A comment left by a user at 9 AM gets processed by the LLM trait extractor, written as structured labels to the database, picked up by the Master on its next training run, baked into a new CTR model, downloaded by Workers, and reflected in that user's recommendation list — all automatically, within the next training interval. The system closes the feedback loop without any manual curation.
The learning cycle is also the system's primary mechanism for handling cold-start users: even if a new user has no interaction history, their first comment produces LLM-extracted style labels that immediately feed into CTR features, giving the ranker something to work with before CF has enough data.
