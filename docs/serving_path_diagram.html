Two Entry Points

  • Anonymous user — provides session items, hits POST /session/recommend
Path A — Hot Path
The normal operating mode. Worker pre-computed everything. Server just reads Redis → filters → returns. Sub-millisecond.
Path B — Graceful Degradation
Falls back to raw CF scores — a separate cache key written by Worker's CF update step. No re-ranking. Self-heals when Worker runs next cycle.
Path C — Only Online Path
The only path with live computation on the request. But all similarity scores are pre-built by Master — so it's still just cache reads + in-memory aggregation.

Gorse Core
Config Sync
Server has no local config file. It pulls the full config JSON from Master on every poll. Change config on Master → all Server nodes converge within one MetaTimeout interval.
SQLite Proxy
If DataStore or CacheStore is SQLite (local file), Server uses a gRPC proxy through Master instead of a direct connection. Transparent to all path logic.
Why Path A is Fast
The Worker did all recall + ranking work offline. The Server's only "computation" is: 1 Redis read + in-memory list filter. Total: <5ms in typical deployments.
Path B Self-Heals
A cache miss does NOT trigger a Worker run. The degraded CF-only response is served until the Worker's next scheduled cycle naturally populates the cache. No manual intervention needed.
Path C as RAG Retrieval
Session items = the query. Item-to-item index = the document store. Score aggregation = the retrieval step. The closest analog to pure RAG in the entire system — no user model involved.
Identical Response Shape
All three paths return the exact same JSON structure. The client has no way to know whether it received a full re-ranked result (Path A), a CF fallback (Path B), or a session-based list (Path C).
The architectural principle: The Server is stateless by design. Every path either reads a pre-populated cache written by the Worker (Paths A & B) or reads a pre-built similarity index written by the Master (Path C). The Server can scale horizontally with no coordination — all shared state lives in CacheClient and DataClient, which every node reads from independently.
