Path A — Warm User · Cache Hit · Most Common
Personalized Recommendation Request — Served from Cache
The user has prior interaction history. The Worker already generated and cached their ranked recommendation list. This is the hot path — the Server does zero computation.
  • 1
    Client sends a recommendation request. The fashion-recommend frontend (or any API consumer) calls GET /recommend/{userId} on the Gorse Server, optionally with a category filter and an n (number of results) parameter.
  • 2
    Server validates the request and resolves the user. It checks that the userId exists, reads any active category filter, and determines the maximum result count. No model is loaded — the Server holds no trained artifacts in memory.
  • 3
    Server reads the pre-computed ranked list from CacheClient. It looks up the key Recommend/{userId} (and optionally Recommend/{userId}/{category}) in the cache store (Redis). This list was written by the Worker during the last offline recommendation generation cycle.
    • Decision point — is the cache populated?
    YES → Cached list exists and has not expired (TTL set by Config.Recommend.CacheExpire) — continue on Path A
    NO → Cache is empty or expired — divert to Path B (Cache Miss fallback)
  • 4
    Server filters out items the user has already seen. Reads the user's feedback history from DataClient. Any item the user has already interacted with (based on Config.Recommend.Read.Exclude feedback types) is stripped from the result list before returning.
  • 5
    Server applies the category filter (if provided). If the request specified a category, only items matching that category are kept. The pre-computed list already contains category metadata per item so this is a simple in-memory filter pass.
  • 6
    Server returns the top-N items as a JSON response. Each item includes its ID, score, categories, and timestamp. The response is the direct output of the offline Worker pipeline — shaped by whatever ranker (FM, LLM, or pass-through) was active when the Worker last ran.
Latency profile
One cache read + one feedback read + in-memory filtering. Sub-millisecond to low-millisecond. No model inference, no external API calls, no database scans.
Why this path exists: The entire architecture is built to make this path as fast as possible. The Worker does all the expensive work offline, asynchronously, for every user on a schedule. The Server's job is simply to be a fast read-through layer on top of the cache. In a healthy, warm system, nearly all recommendation requests take this path.
Path B — Warm User · Cache Miss · Fallback
Personalized Recommendation Request — Real-Time CF Fallback
The user exists and has history, but their recommendation cache is empty or expired. The Server falls back to a direct collaborative filtering lookup — bypassing the full offline pipeline.
  • 1
    Cache miss is detected. The Server queries CacheClient for Recommend/{userId} and gets an empty result — either the user is new to the cache, their TTL has expired, or the Worker hasn't run yet for this user's shard.
  • 2
    Server falls back to Collaborative Filtering scores. It reads from the CollaborativeFiltering/{userId} cache key — a separate cache entry written directly by the Worker's CF update step (not the full ranked pipeline). These scores represent raw CF dot-product similarities without the full recall + rank pipeline.
    • Decision point — do CF scores exist in cache?
    YES → Use CF scores as the recommendation list — no ranking applied
    NO → User is truly cold (no CF embedding exists) — return empty or fall through to non-personalized
  • 3
    Server applies the same post-processing as Path A. Filters out already-seen items, applies category filter if present, truncates to top-N. The result is less refined than Path A — it reflects CF ordering only, with no CTR re-ranking or LLM reranking on top.
  • 4
    Server returns the CF-ordered result. The response shape is identical to Path A — same JSON structure, same fields. The client cannot distinguish a cache hit from a fallback response.
  • ※
    Cache miss does not trigger a Worker run. The Server has no mechanism to kick off a recommendation generation job. The Worker will naturally pick up this user on its next scheduled cycle. The cache miss is self-healing — it resolves automatically.
When this occurs
New deployments (cold cache), users who haven't been active in a while (TTL expired), or users who just registered before the first Worker cycle ran. Also affects users whose Worker shard is temporarily offline.
The tradeoff: Path B is intentionally degraded — it serves CF-only results instead of the full recall + rank output. This is an explicit architectural choice: the Server stays stateless and computation-free. A full real-time re-run of the Worker pipeline on the request path would be too slow and too expensive to do synchronously per request.
Path C — Session-Based · No User ID · Fully Online
Session Recommendation — Real-Time Item-to-Item on the Request Path
No userId is known — an anonymous visitor or a new user in their first session. The client provides a list of items the user has interacted with in this session, and the Server computes recommendations on-the-fly using item-to-item similarity. This is the only fully online path in the system.
  • 1
    Client sends a session recommendation request. Calls POST /session/recommend with a request body containing:
    • A list of feedback items — items the user has interacted with in this session, each tagged with a feedback type (view, like, etc.)
    • An optional category filter
    • An optional n (number of results)
    No userId is required or stored. The session is stateless.
  • 2
    Server classifies the session feedback by type. Splits the provided items into positive interactions (e.g., liked, purchased) and weaker signals (e.g., viewed). Positive interactions carry more weight when driving similarity lookups.
  • 3
    Server performs item-to-item similarity lookups for each session item. For every item the user interacted with, it reads the pre-computed item-to-item similarity scores from CacheClient. This is the key difference from Path A and B — instead of looking up a user's pre-ranked list, it aggregates similarity vectors anchored to the session items.
  • 4
    Server merges and deduplicates the similarity results. Items that appear as similar to multiple session items have their scores summed. This means items that are similar to several things the user engaged with rank higher — a simple but effective form of session-aware scoring.
    • Items already in the session feedback list are excluded from results
    • Scores are sorted descending after aggregation
  • 5
    Server applies category filter and truncates to top-N. Same post-processing as Paths A and B — category filter applied, result capped at requested count.
  • 6
    Server returns the session-based result immediately. No caching of this result — it's ephemeral and tied to this specific session context. The next session request with different items will produce a different result.
Latency profile
N cache reads (one per session item) + in-memory score aggregation + sort. Latency scales with the number of session items provided but remains fast since all item-to-item scores are pre-computed by the Master.
Why this path is architecturally distinct: Paths A and B are fundamentally user-centric — they look up a pre-computed list tied to a known user. Path C is item-centric — it treats the session as a query and the item-to-item index as the retrieval mechanism. No user model, no CF, no CTR model is involved. It's the closest analog to a pure RAG retrieval in the entire system — session items are the query, item-to-item similarity scores are the index.

The architectural principle behind all three paths: The Server is deliberately kept stateless and computation-free. It holds no models, runs no training, and generates no recommendations. Every path either reads from a pre-populated cache (Paths A and B) or reads from a pre-computed index (Path C). This means the Server can scale horizontally without any shared state — adding more Server instances requires no coordination, because all state lives in the shared CacheClient and DataClient that all nodes read from.
